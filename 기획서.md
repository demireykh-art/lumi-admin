# 루미의원 병원 CRM 통합앱 기획서 (초안)

> 작성일: 2026-06-21
> 상태: **초안** — "내가 전달할 병원 정보" 빈칸을 채우면 상세 설계로 확정합니다.
> 목적: 기존 ERP에 **환자 단위 CRM**을 결합해 *환자입력 · 재고분석 · 매출분석 · 환자매출분석 · 지출구조 · 마케팅 의사결정*을 한 앱에서 운영.

---

## 1. 현황 분석 (AS-IS)

기존 코드는 Firebase(Firestore) + 정적 HTML/JS(모듈별 `.js`) 구조의 **단일 병원** 관리 ERP입니다. (`locations`는 별도 지점이 아니라 *5층-처치실 / 6층-시술실* 같은 **원내 보관 위치**)

### 1.1 모듈 현황

| 영역 | 상태 | 핵심 컬렉션 | 비고 |
|---|---|---|---|
| **매출** | ✅ 운영 중 | `revenue`, `salesDetail` | **월 단위 집계만** 존재 (`total, nonInsurance, japan, patients, transactions...`). 의사별/일본인스태프별 집계는 `salesDetail.doctorSales / japanStaffSales`. **환자 개인 단위 데이터 없음** ← 최대 공백 |
| **지출** | ✅ 운영 중 | `expense`, `fixedExpenses`, `variableExpenses`, `cards`, `categoryRules` | 카드내역 파싱·자동 카테고리 분류 보유 |
| **재고** | ✅ 운영 중 | `inventory`, `inventoryLogs`, `inventoryAudits`, `purchaseRequests`, `recipes` | **`recipes`가 시술명→소모품→원가를 이미 연결** (`calculateRecipeCost`). 위치별 재고 지원 |
| **인사/급여** | ✅ 운영 중 | `employees`, `attendance`, `payroll`, `leaveRequests`, `incentive*`, `*Taxes`, `mealRecords`, `lunchOT` | 근태·연차·급여·인센티브·세금 |
| **재무** | ✅ 운영 중 | `finance`, `config`, `settings` | 손익 기초 |
| **권한** | ✅ 기초 | `admins` | 역할 기반 세분화는 미흡 |

### 1.2 핵심 진단
1. **가장 큰 공백 = 환자(고객) 단위 데이터.** 현재는 "이번 달 비보험 매출 X / 환자 Y명"까지만 알 수 있고, 개별 환자의 시술·결제·재방문 이력이 없음. → **환자매출분석·마케팅 ROI는 이 데이터 없이는 불가능.**
2. **유입경로(채널) 데이터 없음.** → 채널별 CAC/ROI 산출 불가.
3. **자산은 충분.** 시술 원가(`recipes`), 지출 분류(`expense-categories`), 재고 로그가 이미 있어 환자 데이터만 채우면 분석이 빠르게 연결됨.

## 2. 목표 (TO-BE)

> "환자 1명을 등록·시술 기록하면 → 매출·재고·원가·마케팅 ROI가 자동으로 흐르는 통합앱"

### 2.1 6대 요구 기능 매핑

| 요구 | 구현 방향 | 의존 데이터 |
|---|---|---|
| ① 원내 환자 내용 입력 | **환자 카드 + 방문/시술 기록** 신규 모듈 (`patients`, `visits`) | 환자 입력 항목·시술 마스터 |
| ② 재고분석 | 기존 `inventory` + **시술기록→`recipes`→자동 소모품 차감/예측** | 시술↔소모품 매핑 정비 |
| ③ 매출분석 | 기존 월집계 + **환자기록 롤업**(시술별/의사별/신환·구환) | `visits` |
| ④ 환자 매출분석 | **LTV·재방문율·전환율·세그먼트** 신규 대시보드 | `patients` + `visits` |
| ⑤ 지출구조 확인 | 기존 `expense` + **매출 대비 손익/비용비중** 통합 뷰 | 비용 분류표 |
| ⑥ 마케팅 의사결정 | **채널별 비용 ↔ 유입환자 ↔ 매출** = CAC/ROAS/LTV | `channels`, 환자 유입경로 |

### 2.2 핵심 지표 (KPI) 후보
- 신환 수 / 재진율 / 휴면환자(예: 6개월+ 미방문)
- 객단가(전체/비보험/시술별), 환자 LTV
- 상담→결제 전환율
- 채널별 CAC(획득단가), ROAS(광고비 대비 매출), 채널 LTV
- 매출총이익률(매출 − 시술원가), 영업이익(− 지출)

## 3. 통합 데이터 모델 (신규 컬렉션 초안)

기존 컬렉션은 유지하고 아래를 추가합니다. (필드는 확정 전 초안)

```
patients (환자 마스터)   ※ 주민번호=민감정보 → §6 보안정책 적용
  { id, name, phone,
    rrnFront,          // 주민번호 앞 6자리 (생년월일 → birth/age 파생)
    rrnGender,         // 뒤 1자리 (성별·내외국인 파생)
    firstVisitAt, channelId,           // 유입경로 → 마케팅 분석 연결
    tags[], grade, memo,               // 세그먼트/VIP, 차트메모
    isJapanese, recallAt, status }     // 휴면/활성

visits (방문·시술 기록, 환자당 N건)  ← 분석의 원천
  { id, patientId, date,
    doctorId,            // 진료(창)
    assistantId,         // 어시스트(창)
    consultantId,        // 상담(창)
    ordererId,           // 오더(창)   ← 4개 담당 모두 employees와 연결
    items: [ { treatmentName, price, discount,
               qty, recipeId } ],       // recipeId로 원가·재고 차감 연결
    paid, payMethod, isNonInsurance,
    nextReservation, consultOnly }      // 상담만/결제 구분 → 전환율

treatments (시술 마스터, 정가표)        ※ 정가표 전달 예정
  { id, name, category, listPrice,
    isNonInsurance, recipeId }          // 원가는 recipes(재고내역) 연결

channels (마케팅 채널)
  { id, name, type,                     // paid | organic
    monthlyCost: { '2026-06': 0 } }
    // 확정: 일본마케팅(paid) 월 500만원 / 블로그·SNS(organic) 0원

(파생) salesDetail 는 visits 롤업으로 자동 생성/대체 검토
```

연결 흐름:
`visits.items.recipeId → recipes → 소모품 자동차감(inventoryLogs) + 원가` ,
`patients.channelId → channels.monthlyCost` 로 마케팅 ROI 계산.

```

2. 환자
   2-1. 환자 목록/검색         (등록·태그·휴면 필터)

   2-3. 방문/시술 입력          (★ 핵심 입력 화면)

8. 인사/급여   (기존 유지)
9. 설정/권한   (역할별 데이터 접근)
```

## 6. 정책·리스크
- **주민번호(고유식별정보) 취급**: 입력하시는 *앞6+뒤1*은 개인정보보호법상 **민감/고유식별정보**. 권장 처리방식 — ① 화면엔 마스킹(예: `900101-1******`) ② 저장 시 **암호화** ③ 분석엔 생년월일·성별·연령대만 **파생값**으로 사용(원본 직접조회 최소화) ④ 조회·수정 **접근로그** 기록.
- **개인정보(환자정보)**: 접근권한 분리·접근로그·최소수집 원칙 필요. 의료법/개인정보보호법 검토.
- 기존 월집계 매출과 신규 환자기록 롤업 간 **정합성**(과거 데이터 마이그레이션 범위) 결정 필요.
